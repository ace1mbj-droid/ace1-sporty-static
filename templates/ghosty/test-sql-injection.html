Common SQL Injection Attack Patterns

Attack 1: Authentication Bypass BLOCKED

Attack Goal: Bypass login without knowing the password

Email: admin' OR '1'='1'-- Password: anything Expected Vulnerable SQL: SELECT * FROM users WHERE email = 'admin' OR '1'='1'--' AND password = 'hash' Result: Returns all users (bypassed!)
Your Supabase Implementation: .eq('email', "admin' OR '1'='1'--") What Happens: - Looks for email = "admin' OR '1'='1'--" (exact match) - No user with that email exists - Login fails ✅ - Attack BLOCKED!

Attack 2: Table Deletion BLOCKED

Attack Goal: Delete the entire users table

Email: admin'; DROP TABLE users;-- Expected Vulnerable SQL: SELECT * FROM users WHERE email = 'admin'; DROP TABLE users;--' Result: Users table deleted (disaster!)
Your Supabase Implementation: .eq('email', "admin'; DROP TABLE users;--") What Happens: - Treats entire string as email value - No SQL commands executed - Table is safe ✅ - Attack BLOCKED!

Attack 3: Data Extraction (UNION) BLOCKED

Attack Goal: Extract all password hashes

Email: admin' UNION SELECT password_hash FROM users-- Expected Vulnerable SQL: SELECT * FROM users WHERE email = 'admin' UNION SELECT password_hash FROM users--' Result: Returns all password hashes (data breach!)
Your Supabase Implementation: .eq('email', "admin' UNION SELECT password_hash FROM users--") What Happens: - Treats as literal email search - No UNION executed - No data leaked ✅ - Attack BLOCKED!

Attack 4: Blind SQL Injection BLOCKED

Attack Goal: Extract data bit by bit using timing

Email: admin' AND SLEEP(5)-- Expected Vulnerable SQL: SELECT * FROM users WHERE email = 'admin' AND SLEEP(5)--' Result: 5 second delay reveals info
Your Supabase Implementation: .eq('email', "admin' AND SLEEP(5)--") What Happens: - No SQL function execution - Immediate response - No timing-based attacks possible ✅ - Attack BLOCKED!
